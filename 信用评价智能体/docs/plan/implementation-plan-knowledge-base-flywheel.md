# 信用评价知识库与数据飞轮实现计划

> 版本：v0.1  
> 日期：2026-03-08  
> 状态：已确认方向，待实施

## 1. 目标与结论

围绕 PRD Phase 8（知识沉淀与入库）建设信用评价知识库，使模型构建从“单次生成”升级为“可持续增强”。

架构结论：**前台一体化、后台模块化**。

- 前台：与现有模型构建工作台整合（同一应用、同一流程）
- 后台：知识库能力独立为模块/服务（便于扩展、治理、替换）

## 2. 架构边界

### 2.1 前台（整合）

在现有 1-7 阶段中嵌入知识库能力：

- 需求阶段：展示“检索命中依据”
- 构建/微调阶段：展示“推荐来源条款/案例”
- 发布阶段：提供“提交入库”操作
- 项目维度：展示“当前模型的知识贡献状态（草稿/待审/入库）”

### 2.2 后台（模块化）

拆分 3 个逻辑模块（可单体实现，后续可拆服务）：

1. `kb-ingest`：文档解析、切片、标签、向量化入库  
2. `kb-retrieval`：检索召回（关键词+向量）与重排  
3. `kb-governance`：脱敏、质量评分、审核流转、发布入库

## 3. 知识对象模型

统一定义 `KnowledgeAsset`（建议先在 TS + 后端 DTO 保持一致）：

- `assetType`: `policy` | `indicator_dict` | `rule_template` | `model_case` | `management_doc`
- `title`
- `source`（上传/检索/系统沉淀）
- `content`（原文或结构化内容）
- `tags`（行业、地区、评分制、等级制、规则类型等）
- `qualityScore`（验算表现/人工评分）
- `status`: `draft` | `pending_review` | `approved` | `published` | `rejected`
- `trace`（来源链路：项目ID、模型版本、解析时间、脱敏策略）

## 4. 数据飞轮闭环

1. 用户完成模型构建与验算  
2. 发布阶段触发“提交入库”  
3. 系统自动执行：脱敏 -> 抽取 -> 标签化 -> 质量评分  
4. 审核通过后写入知识库（结构化 + 向量索引）  
5. 新项目构建时优先检索命中高质量知识并生成推荐  
6. 继续产生新模型，形成迭代增强

## 5. 分阶段实施

## Phase A（MVP，前端可演示）

目标：先跑通流程，形成可见飞轮闭环。

- 新增知识库数据结构（mock）
- 新增“命中依据”展示区
- 新增“提交入库”按钮与状态流转（mock 审核）
- 新增“我的知识贡献”列表
- 对话指令支持：
  - `提交当前模型入库`
  - `查看知识库命中`

验收标准：

- 用户可看到“推荐来自哪条政策/哪份案例”
- 模型发布后可提交入库并看到状态变化
- 新项目能复用之前入库内容

## Phase B（后端接入）

目标：去 mock，接真实 RAG 与治理流程。

- 接入文档解析与向量库
- 接入检索服务（混合检索+重排）
- 接入审核工作流（待审/通过/驳回）
- 保留前端交互不变，只替换数据源

验收标准：

- 检索结果可追溯（来源链路完整）
- 入库内容可审核、可撤回、可版本化
- 质量门槛生效（低质量模型不可自动入库）

## 6. 与现有系统的接口草案

最小接口（建议）：

- `POST /kb/ingest`：提交入库任务
- `GET /kb/assets`：知识列表查询（含状态）
- `POST /kb/retrieve`：按需求检索候选知识
- `POST /kb/review/:id`：审核通过/驳回
- `GET /kb/hits/:projectId`：查看当前项目命中依据

## 7. 风险与控制

- 风险：低质量模型污染知识库  
  - 控制：入库门槛 + 审核流
- 风险：敏感信息泄露  
  - 控制：脱敏策略必经 + 审核前不可发布
- 风险：前后端耦合过深影响演进  
  - 控制：统一接口契约，前台只消费标准 DTO

## 8. 下一步执行建议

按优先级执行：

1. 先落 Phase A（2-3 天可形成演示闭环）
2. 同步定义后端接口契约（OpenAPI 草案）
3. 再接 Phase B 的真实检索与治理能力
